浅析HTTPs的流程和逻辑

Posted on 2020-05-15 In Technical , NetWork

Https的东西，看过几次了，每次看完都觉得很有意思。自己回想时候，又容易忘记一些细节。

总结一下，主要是为了保证Http的传输安全，涉及到的参与方有:

  • 证书提供方
  • 服务端
  • 客户端

涉及到的密钥有

  • 证书机构的非对称密钥
  • 服务器证书的非对称密钥
  • 加密通信内容的对称密钥

简单版传输过程

  1. 客户端发起请求;
  2. 服务端返回带签名的证书公钥;
    • 签名是证书机构的私钥签的;
    • 这样客户端可以根据证书机构的公钥判断传输过来的证书公钥是真的;
  3. 客户端根据本地浏览器的证书机构的公钥，验证证书公钥的签名验证它是真的;
    • 证书机构的公钥为了确保真实性，避免传输的问题，一般是直接内置在浏览器中;
  4. 客户端确保公钥真实性后，用公钥加密pre-master secret 给服务端，并且根据该key(和之前协商的加密组件)生成对称密钥;
  5. 服务端收到客户端的请求后，用私钥解开，获取pre-master secret, 并且根据该key(和之前协商的加密组件)生成对称密钥;
  6. 之后所有的请求使用对称密钥通信。

如图:

逻辑点有:

  1. 使用一堆密钥加密内容，就能保证报文的安全性;
  2. 使用对称密钥，双方协商密钥的过程容易被截取，导致整个传输过程变透明，因此使用非对称密钥; 私钥依然在服务端，只需要给客户端一个公开的公钥; 这样基本就保护了服务端（但是客户端依然有风险）
  3. 承上，如果请求和返回被中间人攻击了，依然有可能导致客户端看到的是假的公钥而感知不到，导致信息被截取; 因此引入了第三方的证书的签名来验证公钥的真实性;
  4. 那如何确保验证签名时候使用的公钥是安全的呢？似乎陷入了一个鸡生蛋、蛋生鸡的问题。最后采用的方案是: 内置可信赖的厂商的公钥证书到浏览器里面。因此只要本机浏览器是正常的，就能确保通信可靠了。（如果本机浏览器都被黑了。。。那还说个毛线通信安全了，烂透了都）
  5. 非对称加密更消耗资源，因此只用非对称加密传输对称密钥，最后请求的加解密使用对称密钥完成;

传输步骤详细版过程

摘自《图解HTTP协议》

  • 步骤1： 客户端通过发送Client Hello报文开始SSL通信。报文中包含客户端支持的SSL的指定版本、加密组件（Cipher Suite）列表（所使用的加密算法及密钥长度等）。
  • 步骤2： 服务器可进行SSL通信时，会以Server Hello报文作为应答。和客户端一样，在报文中包含SSL版本以及加密组件。服务器的加密组件内容是从接收到的客户端加密组件内筛选出来的。
  • 步骤3： 之后服务器发送Certificate报文。报文中包含公开密钥证书。
  • 步骤4： 最后服务器发送Server Hello Done报文通知客户端，最初阶段的SSL握手协商部分结束。
  • 步骤5: SSL第一次握手结束之后，客户端以Client Key Exchange报文作为回应。报文中包含通信加密中使用的一种被称为Pre-master secret的随机密码串。该报文已用步骤3中的公开密钥进行加密。
  • 步骤6： 接着客户端继续发送Change Cipher Spec报文。该报文会提示服务器，在此报文之后的通信会采用Pre-master secret密钥加密。
  • 步骤7： 客户端发送Finished报文。该报文包含连接至今全部报文的整体校验值。这次握手协商是否能够成功，要以服务器是否能够正确解密该报文作为判定标准。
  • 步骤8： 服务器同样发送Change Cipher Spec报文。
  • 步骤9： 服务器同样发送Finished报文。
  • 步骤10： 服务器和客户端的Finished报文交换完毕之后，SSL连接就算建立完成。当然，通信会受到SSL的保护。从此处开始进行应用层协议的通信，即发送HTTP请求。
  • 步骤11： 应用层协议通信，即发送HTTP响应。
  • 步骤12： 最后由客户端断开连接。断开连接时，发送close_notify报文。上图做了一些省略，这步之后再发送TCP FIN报文来关闭与TCP的通信。

在以上流程中，应用层发送数据时会附加一种叫做MAC（Message Authentication Code）的报文摘要。MAC能够查知报文是否遭到篡改，从而保护报文的完整性。

一致性哈希算法介绍

Posted on 2020-04-26 Edited on 2020-04-27 In Technical , Distributed System

什么是一致性哈希算法？

一种特殊的hash算法，主要解决了简单哈希算法在分布式哈希表( Distributed Hash Table，DHT) 中存在的动态伸缩等问题。
一致性hash的使用在分布式系统的比较常见，比如分布式存储，分布式计算等等（多队列多任务）

为什么要用一致性哈希算法？

以缓存集群为例，为了提高缓存的吞吐量，我们通常会把缓存求hash然后分配到不同的节点上。
我们要分配节点第一想到的办法就是取模。即现在有3个节点，假设资源key=7,7%3=1，则选择Node1，key=3，3%3=0，则选择Node0。
这种方法虽然简单，但有个缺点，如果节点数增加或减少，就会有大量的key不命中，请求会直接到达数据库（击穿）。

类似的场景还有rpc服务的负载均衡。为了解决这种问题，让节点数量变化时候，只会引起很少部分的key不会命中，于是有了一致性hash算法。

逻辑

原理解析

前面提到，取余算法会有大量的key不命中，那么如何减少不命中的key呢？

其实原理很简单，就是分区域。
首先，hash值作为一个整型，会落在 0 ~ 2^32 -1 的范围内。
如果我有4个节点，那么把hash值划分为4个区域，0 ~ 2^30 -1，2^30 ~ 2^31 -1，2^31 ~ 3 * 2^30 -1，3 * 2^30 ~ 2^32 -1

  • 当我再有一个新节点加入时候，只需要把其中任何一截拆成两小区域，那么就只有其中一小区域的key无法命中。
  • 当我任何一个节点被移除（或者宕机），只需要把相邻的两个区域合并，那么就只有其中一个区域的key无法命中。

有很多相关文章具体表现上通常喜欢用环来表示，表述如下:

  1. 求出服务器（节点）的哈希值，并将其配置到0～232的圆（continuum）上。
  2. 采用同样的方法求出存储数据的键的哈希值，并映射到相同的圆上。
  3. 从数据映射到的位置开始顺时针查找，将数据保存到找到的第一个服务器上。如果超过232仍然找不到服务器，就会保存到第一台服务器上。

如果直接上面这样分段的话，新的一小段只有旧的一般，如何让分段更加的均匀呢？
另外如果总是相邻节点宕机，会不会让一个节点的负载变的很大？
具体实现上，通常会采用虚拟节点的办法。

虚拟节点

承上，虚拟节点主要想做的，是让各个节点分配的资源尽量均匀。
因此通常会把一个节点映射成多个虚拟节点:

  • 加入单个新节点时候，也会被映射成多个节点，因此分段更均匀，移除时候也更均匀
  • 避免分配时候，由于hash算法，或者分段规则等原因，导致各个节点之间分配的区域大小差别很大

一般如果节点数越少，每个节点对应的虚拟节点数就越多。

如何给开源项目提交代码

Posted on 2020-04-22

步骤

只需要follow以下几步：

  1. fork 想要贡献的项目repository。
    原项目一般称之为’original repository’ 或者 ‘upstream repository’; 这个只需要到原项目点一下就行
  2. 本地clone 自己fork的代码库
  3. 给本地clone的项目添加upstream。记得任何时候及时同步(sync)自己的fork 与 upstream repository
  4. 编辑时候，到自己的fork repository 编辑
  5. 同步upstream repository 并提交
  6. 去自己的github 的fork repository上，给upstream repository提交pr

这里涉及的其实主要的就是github的fork功能。

什么是github fork?

经常用于github的开源项目开发的一个特性。类似于copy代码，不过提供了两个功能:

  1. 提交pr: 你可以使用pr，来从你自己的fork向original(upstream) repository 提交修改申请
  2. 同步最新代码: 你可以把upstream repository的更新同步到你自己的fork上，来保持与开源项目的同步

如何使用fork?

1. fork代码，本地clone并添加upstream

  1. fork代码去original repository 点两下;
  2. 本地git clone ${FORK-REPOSITORY}
  3. 进入项目目录查看本地remote。
  4. 添加upstream，并查看

    2. fork仓库与upstream 同步

  5. 从upstream repository 拉代码
  6. 切换到本地分支, 并同步最新upstream代码

3. 修改代码，再次同步，并提交pr

  1. 在自己的fork修改代码并提交
  2. 同步代码
  3. 去自己的fork repository里面提交pr

如何快速启动一个项目 -- inception 方法论简介

Posted on 2020-04-17 Edited on 2020-04-26 In Project , 业务分析 , 方法论

Inception算是在公司接触比较多的、项目启动时候的一套模板方法论。

什么是Inception – 项目启动的一个实践方法论

启动项目的方法。
用集中、互动的工作方式，通过3天到2周的时间，帮助客户对未来一个季度到一年产品方案、项目交付计划达成一致，能快速进入项目交付环节。

为什么要Inception – 快速启动，及时响应市场变化

Inception解决的项目启动的问题。因此Inception之前，希望客户有一个明确的想解决的问题，或者一个明确的一个产品想法。

inception通过快速启动来响应市场的变化。
那么inception相对于别的方法，是如何确保快速启动同时，不会降低交付价值的？

在我理解，Inception启动时候的核心关注点在于:

  • 响应变化: 变化永远存在，因此项目从idea到产品越快越好，也不宜对于未来的东西做过度设计；
  • 达成一致: 确保交付团队跟业务人员对于目标、交付物等达成一致；交付团队能够明白:
    • 这个项目背景是什么？
    • 这些业务需求是如何确定下来的？
    • 优先级是什么？
    • 负责人最关注的是什么？
    • 风险在哪？
  • 可视化: 流程结果的可视化; 客户能够明白:
    • 产品会做成什么样？质量如何？
    • 花多久？产品演进路线如何？什么时候能看到什么东西

因此，最后的目标产出物，应当包括:

  • 产品愿景。当大家对产品的目标愿景理解一致，才能在一致的背景下设计产品;
  • 产品蓝图。对于产品有个大致的规划，能够支撑产品交付范围的优先级判断;
  • 产品设计。业务上和技术上的high level设计，比如业务上的线框图，技术上的架构图;
  • 交付范围
  • 交付计划

如何Inception

以典型的业务场景为例，大致流程如下:

tips:

  1. 每个RM使用DataSourceProxy连接数据库，其目的是使用ConnectionProxy，使用数据源和数据连接代理的目的就是在第一阶段将undo_log和业务数据放在一个本地事务提交，这样就保存了只要有业务操作就一定有undo_log。
  2. 在第一阶段undo_log中存放了数据修改前和修改后的值，为事务回滚作好准备，所以第一阶段完成就已经将分支事务提交，也就释放了锁资源。
  3. TM开启全局事务开始，将XID全局事务id放在事务上下文中，通过feign调用也将XID传入下游分支事务，每个 分支事务将自己的Branch ID分支事务ID与XID关联。
  4. 第二阶段全局事务提交，TC会通知各各分支参与者提交分支事务，在第一阶段就已经提交了分支事务，这里各 各参与者只需要删除undo_log即可，并且可以异步执行，第二阶段很快可以完成。
  5. 第二阶段全局事务回滚，TC会通知各各分支参与者回滚分支事务，通过 XID 和 Branch ID 找到相应的回滚日 志，通过回滚日志生成反向的 SQL 并执行，以完成分支事务回滚到之前的状态，如果回滚失败则会重试回滚操 作。

  • 李四事务失败，张三事务回滚成功。
  • 张三事务失败，李四事务回滚成功。
  • 分支事务超时测试。

Java中的四种引用类型

Posted on 2020-04-07 Edited on 2020-05-15 In Technical , Java , JVM

Java中的引用是什么

JDK1.2之前，引用指的是: 如果reference类型的数据中，存储的数值代表的是另一块内存的起始地址，就称该reference数据是代表某块内存、某个对象的引用。
这个定义下只有“被引用”和“未被引用” 两种状态。不是很灵活，比如缓存一般期望的是: 内存空间充足，保存在内存中；内存空间不够，就被回收。

因此在JDK1.2之后，Java扩展了引用，有四种引用类型:

  • 强引用（Strongly Reference）
  • 软引用（Soft Reference）
  • 弱引用（Weak Reference）
  • 虚引用（Phantom Reference）

四种引用类型

大概的对比如下:

类型 回收时间 使用场景
强引用 不回收 最常见的日常场景
软引用 即将OOM时候 经常使用的缓存，避免OOM问题，图片、视频编辑
弱引用 垃圾回收时候 使用概率较低缓存，WeakHashMap
虚引用 无法访问，引用被回收时候回收对象 运行对象销毁前的一些操作；
比finalizer()更可靠，更推荐

强引用（Strongly Reference）

传统的常见“引用”。比如Object o = new Object()，只要存在，就不会被回收。

软引用（Soft Reference）

还有用，但非必须的对象，在OOM之前会把他们回收，如果这些回收完了还是内存不够，才会OOM。可以通过java.lang.ref.SoftReference使用软引用。
软引用可以和一个引用队列（ReferenceQueue）联合使用，如果软引用所引用的对象被垃圾回收，Java虚拟机就会把这个软引用加入到与之关联的引用队列中。

使用场景:
通常用于内存敏感的高速缓存。此外图片编辑器，视频编辑器之类的软件可以使用这种思路。

弱引用（Weak Reference）

还有用，但非必须的对象。下次垃圾收集会被回收。

使用场景：
弱引用和软引用都可以用来做缓存，可以根据情况来判断使用哪一种缓存。
弱引用也通常用于实现规范化映射。

虚引用（Phantom Reference）

虚引用的存在对生存时间没有任何影响，也无法通过虚引用获取对象实例，唯一的作用就是对象被回收时候收到一个系统通知。必须配合引用队列（ReferenceQueue）使用。

使用场景：
对象销毁前的一些操作，比如说资源释放等(推荐的做法)。Object.finalize()虽然也可以做这类动作，但是这个方式即不安全又低效，在JDK 9已经被标记废弃。
继承自java.lang.ref.PhantomReference的jdk.internal.ref.Cleaner，是一个更加轻量、健壮的Object.finalize()的替代品。

Java gc综述（包括ZGC）

GC 主要是对Java堆内存进行空间回收。

GC历史与特征

  • Serial + Serial Old
    • 整个GC都STW。
    • 单线程，串行
    • 分代：新生代标记-复制；老年代标记整理
  • Parallel Scavenge + Parallel old。
    • 整个GC都STW
    • 关注点在吞吐量
    • 并行;
    • 分代：新生代标记-复制；老年代标记整理
  • ParNew + CMS.
    • Reference图遍历阶段（并发标记）、与标记完的清理（并发清理）不STW
    • 有浮动垃圾，在堆满之前就要GC（默认92%）
    • CMS失败（垃圾收集过程中堆就满了）会使用Serial Old作为预案
    • CMS使用标记-清除，内存碎片多
    • 增量更新，避免标记过程中的引用变化
  • G1.
    • 并发标记时候不STW
    • 整理时候使用标记-整理算法，需要移动对象，STW
    • 分治的开始。开创面向局部收集的思路 + 基于Region内存布局
    • 面向CSet回收，而不是分代回收
    • GCRoot标记停顿时间短、可控。使用Remember Set避免全堆GCRoot扫描，因此也会耗费额外的内存。写屏障维护Remember Set。
    • 原始快照(SATB)，避免标记过程中的引用变化
    • 同样G1失败（垃圾收集过程中堆就满了）会使用Serial Old作为预案
  • ZGC.
    • STW 10ms 以内；处理16TB内存; 当前最先进的垃圾收集器
    • 核心: 并发整理的实现。Concurrent Relocate, 使用染色指针与读屏障技术，解决了标记-整理过程中的STW
    • 所有流程都可以跟用户进程并发执行，只有之间有很短的STW。初始标记root、最终标记、准备Relocate会有短暂STW
    • 每次回收扫描所有的region, 减少remember set的维护成本
    • 标记时候，标记在染色指针上，不是对象上
    • Region布局，延续了分治的思想
    • 未分代，因为优先级的问题而不是算法的设计。因此如果大量创建朝生熄灭的对象性能会降低
    • Partial Compaction: 把要回收的region有用的对象复制过来。然后直接把那个region干掉。因为染色指针回收区域效率更高。
    • 支持NUMA的CPU结构，CPU分配内存时候，优先分配离得近的内存

其他算法细节

  • 根节点枚举:
    • 所有垃圾收集器，包括ZGC, 根节点枚举都要STW
    • OopMap(Ordinary Object Pointer Map)，支持快速准确完成GC root 枚举。标注出栈帧中的局部变量表哪些变量是引用，这样GC扫描时候可以直接获取这些信息，而不是去到处找
  • 安全点: Safepoint
    • OopMap变化很多，全纪录消耗很大，只在特定地方记录了，称为Safe point
    • 只有到达安全点才能开始GC
    • 抢先式中断与主动式中断（常用）
  • 安全区域
    • 确保某一块区域中，引用关系不会发生变化。比如线程中断或者挂起的地方
  • 记忆集（Remember Set）
    • 缩减GC Root扫描范围，解决跨代/跨区域引用，避免一次把所有对象加入GC Root
    • 卡表是Remember Set的一种实现
  • 写屏障
    • 解决卡表元素维护的问题
    • 区别 “读屏障” 与 解决并发乱序执行的 “内存屏障”概念
  • 并发可达性分析
    • 可达性分析中，用户线程并发执行，避免把存活对象标死了
    • 三色标记法: 黑色扫描完了的，灰色扫描了一半的，白色没扫描的. 如果黑的加了个白的，就会被漏掉
    • 增量更新: 黑色对象加了对白色对象的引用后，就会变成黑的
    • 原始快照: 按照刚开始扫描时候的对象快照图来搜索。

MySQL的InnoDB如何实现事务性

Posted on 2020-04-01 Edited on 2020-04-11 In Technical , MySQL

事务是一个序列操作，其中的操作要么都执行，要么都不执行，它是一个不可分割的工作单位；数据库事务的 ACID 四大特性是事务的基础，了解了 ACID 是如何实现的，我们也就清楚了事务的实现。

MySQL 如何保证事务性?

MySQL事务在哪实现的？

  • MySQL中服务器层不管理事务，事务是由存储引擎实现的。
  • MySQL支持事务的存储引擎有InnoDB、NDB Cluster等，其中InnoDB的使用最为广泛；其他存储引擎不支持事务，如MyIsam、Memory等。

MySQL事务如何实现的？

  • redo log重做日志用来保证事务的持久性
  • undo log回滚日志保证事务的原子性
  • undo log+redo log保证事务的一致性
  • 锁（共享、排他）用来保证事务的隔离性

持久性与redo log

持久性

而事务的持久性就体现在，一旦事务被提交，那么数据一定会被写入到数据库中并持久存储起来。就算数据没来得及写入磁盘就宕机时，在下次重新启动后能够成功恢复数据（持久性）；
当事务已经被提交之后，就无法再次回滚了，唯一能够撤回已经提交的事务的方式就是创建一个相反的事务对原操作进行『补偿』，这也是事务持久性的体现之一。

redo log存在的背景

InnoDB作为MySQL的存储引擎，数据是存放在磁盘中的，但如果每次读写数据都需要磁盘IO，效率会很低。为此，InnoDB提供了缓存(Buffer Pool)，Buffer Pool中包含了磁盘中部分数据页的映射，作为访问数据库的缓冲：

  • 当从数据库读取数据时，会首先从Buffer Pool中读取，如果Buffer Pool中没有，则从磁盘读取后放入Buffer Pool；
  • 当向数据库写入数据时，会首先写入Buffer Pool，Buffer Pool中修改的数据会定期刷新到磁盘中（这一过程称为刷脏）。

Buffer Pool的使用大大提高了读写数据的效率，但是也带了新的问题：如果MySQL宕机，而此时Buffer Pool中修改的数据还没有刷新到磁盘，就会导致数据的丢失，事务的持久性无法保证。

redo log

重做日志 redo log 分为两部分：

  • 内存中的重做日志缓冲（redo log buffer），是易丢失的；
  • 重做日志文件（redo log file），是持久的。InnoDB通过Force Log at Commit机制来实现持久性，即WAL(Write-Ahead Log)，当commit时，必须先将事务的所有日志写到重做日志文件进行持久化，待commit操作完成才算完成。

InnoDB在下面情况下会将重做日志缓冲的内容写入重做日志文件：

  • master thread 每一秒将重做日志缓冲刷新到重做日志文件;
  • 每个事务提交时
  • 当重做日志缓冲池剩余空间小于1/2时

为了确保每次日志都写入重做日志文件，在每次将日志缓冲写入重做日志文件后，InnoDB存储引擎都需要调用一次fsync（刷盘）操作。但这也不是绝对的。用户可以通过修改innodbflushlogattrx_commoit参数来控制重做日志刷新到磁盘的策略，这个可以作为大量事务提交时的优化点。

  • 1参数默认值，表示事务提交时必须调用一次fsync操作。
  • 0表示事务提交时，重做日志缓存并不立即写入重做日志文件，而是随着Master Thread的间隔进行fsync操作。
  • 2表示事务提交时将重做日志写入重做日志文件，但仅写入文件系统的缓存中，不进行fsync操作。

fsync的效率取决于磁盘的性能，因此磁盘的性能决定了事务提交的性能，也就是数据库的性能。所以如果有人问你如何优化Mysql数据库的时候别忘了有硬件这一条，让他们提升硬盘配置，换SSD固态硬盘。重做日志都是以512字节进行存储的，称之为重做日志块，与磁盘扇区大小一致，这意味着重做日志的写入可以保证原子性，不需要doublewrite技术，也不会由于机器断电导致重做日志仅写入一半并留下脏数据。
它有以下3个特性：

  • 重做日志是在InnoDB层产生的
  • 重做日志是物理格式日志，记录的是对每个页的修改
  • 重做日志在事务进行中不断被写入，而且是顺序写入（MySQL持久化时候）

redo log也需要在事务提交时将日志写入磁盘，为什么它比直接将Buffer Pool中修改的数据写入磁盘(即刷脏)要快呢？主要有以下两方面的原因：

  1. 刷脏是随机IO，因为每次修改的数据位置随机，但写redo log是追加操作，属于顺序IO。
  2. 刷脏是以数据页（Page）为单位的，MySQL默认页大小是16KB，一个Page上一个小修改都要整页写入；而redo log中只包含真正需要写入的部分，无效IO大大减少。

redo log 与 bin log

我们知道，在MySQL中还存在binlog(二进制日志)也可以记录写操作并用于数据的恢复，但二者是有着根本的不同的：

  1. 作用不同：redo log是用于crash recovery的，保证MySQL宕机也不会影响持久性；binlog是用于point-in-time recovery的，保证服务器可以基于时间点恢复数据，此外binlog还用于主从复制。
  2. 层次不同：redo log是InnoDB存储引擎实现的，而binlog是MySQL的服务器层(可以参考文章前面对MySQL逻辑架构的介绍)实现的，同时支持InnoDB和其他存储引擎。
  3. 内容不同：redo log是物理日志，内容基于磁盘的Page；binlog的内容是二进制的，根据binlog_format参数的不同，可能基于sql语句、基于数据本身或者二者的混合。

原子性与undo log

原子性

原子性在并发编程里面也非常常见。完成一件事情可能需要多个操作，当这些操作某一步出现错误或抛出异常时，整个操作就可能不会继续执行下去，而已经进行的操作造成的副作用就可能造成数据更新的丢失或者错误。

事务其实和一个操作没有什么太大的区别，它是一系列的数据库操作（可以理解为 SQL）的集合，如果事务不具备原子性，那么就没办法保证同一个事务中的所有操作都被执行或者未被执行了，整个数据库系统就既不可用也不可信。

undo log

为了满足事务的原子性，所有事务进行的修改都会先记录到undo log中，然后进行数据的修改。如果出现了错误或者用户执行了 ROLLBACK语句，系统可以利用Undo Log中的备份将数据恢复到事务开始之前的状态。undo log实现多版本并发控制（MVCC）来辅助保证事务的隔离性。

回滚日志除了能够在发生错误或者用户执行 ROLLBACK 时提供回滚相关的信息，它还能够在整个系统发生崩溃、数据库进程直接被杀死后，当用户再次启动数据库进程时，还能够立刻通过查询回滚日志将之前未完成的事务进行回滚，这也就需要回滚日志必须先于数据持久化到磁盘上，是我们需要先写日志后写数据库的主要原因。

回滚日志并不能将数据库物理地恢复到执行语句或者事务之前的样子。回滚日志不同于重做日志，它是逻辑日志，对数据库的修改都逻辑的取消了。当事务回滚时，它实际上做的是与先前相反的工作。
对于每个INSERT，InnoDB存储引擎都会完成一个DELETE；
对于每个UPDATE，InnoDB存储引擎都会执行一个相反的UPDATE。

事务提交后并不能马上删除undo log，这是因为可能还有其他事务需要通过undo log 来得到行记录之前的版本。故事务提交时将undo log 放入一个链表中，是否可以删除undo log 根据操作不同分以下2种情况：

  • Insert undo log： insert操作的记录，只对事务本身可见，对其他事务不可见(这是事务隔离性的要求),故该undo log可以在事务提交后直接删除。不需要进行 purge操作。
  • update undo log：记录的是对 delete和 update操作产生的 undo log。该undo log可能需要提供MVCC机制，因此不能在事务提交时就进行删除。提交时放入undo log链表,等待 purge线程进行最后的删除。

一致性与redo+undo log

数据库领域其实包含两个一致性:

  • ACID 中的一致性。
  • CAP 定义中的一致性。

这两个数据库的一致性说的完全不是一个事情，很多很多人都对这两者的概念有非常深的误解，当我们在讨论数据库的一致性时，一定要清楚上下文的语义是什么，尽量明确的问出我们要讨论的到底是 ACID 中的一致性还是 CAP 中的一致性。

ACID中，如果一个事务原子地在一个一致地数据库中独立运行，那么在它执行之后，数据库的状态一定是一致的。

  • 第一层意思就是对于数据完整性的约束。我们可以将事务理解成一个函数，它接受一个外界的 SQL 输入和一个一致的数据库，它一定会返回一个一致的数据库。
  • 第二层意思其实是指逻辑上的对于开发者的要求，我们要在代码中写出正确的事务逻辑，比如银行转账，事务中的逻辑不可能只扣钱或者只加钱，这是应用层面上对于数据库一致性的要求。

CAP 定理中的数据一致性，其实是说分布式系统中的各个节点中对于同一数据的拷贝有着相同的值。

通过原子性 + 持久性，就能保证数据库的一致性。

隔离性

隔离性追求的是并发情形下事务之间互不干扰。那么隔离性的实现主要通过锁实现

一个事务可以读取另一个已提交的事务，多次读取会造成不一样的结果，此现象称为不可重复读问题。

该隔离级别是 MySQL InnoDB 默认的隔离级别。

SERIALIZABLE（串行化）

在该隔离级别下事务都是串行顺序执行的，MySQL 数据库的 InnoDB 引擎会给读操作隐式加一把读共享锁，从而避免了脏读、不可重读复读和幻读问题。

  1. 准备两个终端，在此命名为 mysql 终端 1 和 mysql 终端 2，分别登入 mysql，准备一张测试表 test 并调整隔离级别为 SERIALIZABLE，任意一个终端执行即可。
  2. 登录 mysql 终端 1，开启一个事务，并写入一条数据。
  3. 登录 mysql 终端 2，开启一个事务。
  4. 立马切换到 mysql 终端 1,提交事务。 一旦事务提交，msyql 终端 2 会立马返回 ID 为 1 的记录，否则会一直卡住，直到超时，其中超时参数是由 innodb_lock_wait_timeout 控制。

由于每条 select 语句都会加锁，所以该隔离级别的数据库并发能力最弱，但是有些资料表明该结论也不一定对，如果感兴趣，您可以自行做个压力测试。

锁

InnoDB主要有2种锁：行级锁，意向锁(表锁)

行级锁：

  • 共享锁（读锁 S），允许事务读一行数据。事务拿到某一行记录的共享S锁，才可以读取这一行，并阻止别的事务对其添加X锁。共享锁的目的是提高读读并发。
    可以使用 SQL 语句 select * from tableName where … lock in share mode;手动加 S 锁。
  • 排它锁（写锁 X），允许事务删除一行数据或者更新一行数据。事务拿到某一行记录的排它X锁，才可以修改或者删除这一行。排他锁的目的是为了保证数据的一致性。
    可以使用 SQL 语句 select * from tableName where … for update; 手动加 X 锁。

行级锁中，除了S和S兼容，其他都不兼容。

意向锁：

  • 意向共享锁（读锁 IS ），事务想要获取一张表的几行数据的共享锁，事务在给一个数据行加共享锁前必须先取得该表的IS锁。
  • 意向排他锁（写锁 IX），事务想要获取一张表中几行数据的排它锁，事务在给一个数据行加排他锁前必须先取得该表的IX锁。解释一下意向锁

意向锁的主要用途是为了表达某个事务正在锁定一行或者将要锁定一行数据。
e.g:事务A要对一行记录r进行上X锁，那么InnoDB会先申请表的IX锁，再锁定记录r的X锁。在事务A完成之前，事务B想要来个全表操作，此时直接在表级别的IX就告诉事务B需要等待而不需要在表上判断每一行是否有锁。
意向排它锁存在的价值在于节约InnoDB对于锁的定位和处理性能。另外注意了，除了全表扫描以外意向锁都不会阻塞。

锁的算法

InnoDB有三种行锁的算法：

  • Index-row locks：锁单行记录；
  • Gap Lock：间隙锁，锁定一个范围
  • Next-Key Lock：Index Lock加上Index之前的Gap Lock。主要解决的问题是REPEATABLE READ隔离级别下的幻读。 （比如数据库有id=1，10，100的记录，查找id>5的记录，会返回id=10, 100的记录，同时锁住index在(1, +∞)的记录）

这里主要讲一下 Next-Key Lock，利用Next-key Lock锁定的不是单个值而是一个范围，他的目的就是为了阻止多个事务将记录插入到同一范围内从而导致幻读。
但是，如果走唯一索引，那么Next-Key Lock会降级为Record Lock，即仅锁住索引本身，而不是范围。也就是说Next-Key Lock前置条件为事务隔离级别为RR且查询的是结果集, 索引走的非唯一索引、主键索引。

下面我们用个例子详细说一下。首先建立一张表：

事务A执行如下语句：
SELECT * FROM T WHERE f_id = 3 FOR UPDATE; -- 此处f_id是索引，但是非唯一索引

这时SQL语句走非唯一索引，因此使用Next-Key Locking加锁，并且有2个索引，其需要分别进行锁定。对于聚集索引，其仅对id等于5的索引加上Record Lock。而对于辅助索引，其加上Next-Key Lock，锁定了范围（1,3），特别需要注意的是，InnoDB存储引擎还会对辅助索引下一个键值加上Gap Lock，即范围（3.6）的锁。所以如果在新session中执行如下语句都会报错[Err] 1205 - Lock wait timeout exceeded; try restarting transaction：

原理：

  • 对于事务A, SELECT * FROM T WHERE f_id = 3 FOR UPDATE;语句，对于找到的(此处是f_id =3)的记录设置了排它锁, 也就是一个Index-row locks;
  • 同时，Next-Key Lock 会添加Gap Lock, 锁住f_id在(1, 3), (3, 6)范围内的记录

死锁

死锁是指两个或两个以上的进程在执行过程中，由于竞争资源或者由于彼此通信而造成的一种阻塞的现象，若无外力作用，它们都将无法推进下去。此时称系统处于死锁状态或系统产生了死锁，这些永远在互相等待的进程称为死锁进程。

InnoDB 引擎采取的是 wait-for graph 等待图的方法来自动检测死锁，如果发现死锁会自动回滚一个事务。

下面我们通过一个示例来了解死锁。

  1. 准备两个终端，在此命名为 mysql 终端 1 和 mysql 终端 2，分别登入 mysql，再准备一张测试表 test 写入两条测试数据，并调整隔离级别为 SERIALIZABLE，任意一个终端执行即可。
  2. 登录 mysql 终端 1，开启一个事务，手动给 ID 为 1 的记录加 X 锁。
  3. 登录 mysql 终端 2，开启一个事务，手动给 ID 为 2 的记录加 X 锁。
  4. 切换到 mysql 终端 1，手动给 ID 为 2 的记录加 X 锁，此时会一直卡住，因为此时在等待第 3 步中 X 锁的释放，直到超时，超时时间由 innodb_lock_wait_timeout 控制。
  5. 在锁超时前立刻切换到 mysql 终端 2，手动给 ID 为 1 的记录加 X 锁，此时又会等待第 2 步中 X 所的释放，两个终端都在等待资源的释放，所以 InnoDB 引擎会立马检测到死锁产生，自动回滚一个事务，以防止死锁一直占用资源。

此时，通过 show engine innodb status\G 命令可以看到 LATEST DETECTED DEADLOCK 相关信息，即表明有死锁发生；
或者通过配置 innodb_print_all_deadlocks（MySQL 5.6.2 版本开始提供）参数为 ON 将死锁相关信息打印到 MySQL 的错误日志。

锁的优化建议

锁如果利用不好，会给业务造成大量的卡顿现象，在了解了锁相关的一些知识点后，我们可以有意识的去避免锁带来的一些问题。

  1. 合理设计索引，让 InnoDB 在索引键上面加锁的时候尽可能准确，尽可能的缩小锁定范围，避免造成不必要的锁定而影响其他 Query 的执行。
  2. 尽可能减少基于范围的数据检索过滤条件，避免因为间隙锁带来的负面影响而锁定了不该锁定的记录。
  3. 尽量控制事务的大小，减少锁定的资源量和锁定时间长度。
  4. 在业务环境允许的情况下，尽量使用较低级别的事务隔离，以减少 MySQL 因为实现事务隔离级别所带来的附加成本。

为什么有这个改动？char -> byte

主要还是节省空间。

JDK9 之前的库的 String 类的实现使用了 char 数组来存放字符串，char 占用16位，即两字节。

这种情况下，如果我们要存储字符A，则为0x00 0x41，此时前面的一个字节空间浪费了。但如果保存中文字符则不存在浪费的情况，也就是说如果保存 ISO-8859-1 编码内的字符则浪费，之外的字符则不会浪费。

而 JDK9 后 String 类的实现使用了 byte 数组存放字符串，每个 byte 占用8位，即1字节。
private final byte[] value
但是如果遇到ISO-8859-1 编码外的字符串呢？比如中文咋办？

Java9之后String的新属性，coder编码格式

coder 属性默认有 0 和 1 两个值。如果 String判断字符串只包含了 Latin-1，则 coder 属性值为 0 ，反之则为 1
